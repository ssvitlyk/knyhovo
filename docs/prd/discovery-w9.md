# PRD: W9a Discovery Foundation (covers + опис + author shelf)

> Статус: **Затверджено — implementation-ready.** Технологічний стек: **Next.js (App Router)** + **Fastify** +
> **Prisma/PostgreSQL**. W9a — це **discovery-шар даних поверх уже реалізованих** заморожених екранів
> (Search Results v1.0, Book Details v1.1 / Variant C, «Бажанки» v1.0). Сторінки, BookCard, ліва/права
> панелі Book Details, рядки «Бажанок», типографіка, кольори, відступи **не редизайняться** — W9a лише
> наповнює дані, яких система фізично не мала. Джерело UX — `packages/web/design-import/discovery-w9/`.
> Джерело scope — затверджений аналітичний план `docs/prd/discovery-w9-plan.md`.
>
> **Правило вирішення конфліктів:** коли цей PRD суперечить плану чи дизайн-специфікації — перемагає PRD
> щодо **scope**. Рендеринг Author Shelf визначає **заморожений дизайн-пакет** (§12) — `Author Shelf v1.0`,
> frozen 2026-06-20. Перевага меншому scope, мінімуму бекенду, graceful degradation і прихованню недоступних
> станів замість додаткової роботи. Для вилучених зі scope фіч **не додаються замінні фічі**.

W9 «Discovery» розбито на **три зрізи**; цей PRD охоплює **лише W9a**:

| Зріз | Назва | Статус у цьому PRD |
|------|-------|--------------------|
| **W9a** | **Discovery Foundation** — covers + опис + author shelf | **Повний scope (цей документ)** |
| W9b | Identity & Relationships — метадані, серії, author-сутність | Лише прев'ю (§17) |
| W9c | Popularity & Recommendations — популярність, ML-рекомендації | Лише прев'ю (§18) |

## 1. Overview

W9a — це **Discovery Foundation**, а не **Discovery Destinations**. Він не додає жодної окремої
discovery-сторінки (ні жанрів, ні трендів, ні популярних книг/авторів, ні лендингів, ні рекомендаційної
стрічки). Натомість він закладає **три передумови**, без яких будь-який майбутній discovery-напрямок не має
на чому стояти: **побачити** (обкладинки), **оцінити** (опис), **перейти** (author shelf).

W9a поділений на **два чіткі шари** (той самий принцип, що й W7a — інтелект поверх заморожених патернів):

- **Шар 1 — Discovery Data Foundation (бекенд).** Збагачує модель книги двома полями, яких система фізично
  не мала: обкладинка (`coverUrl`) та опис (`description`). Це перцептивний субстрат дискавері — без
  зображення список книг можна лише *читати*, а не *проглядати*.
- **Шар 2 — Discovery UX Layer (фронтенд).** Перетворює нові дані на discovery-поведінку в **уже наявних**
  екранах: реальні обкладинки в Search/Book Details/«Бажанках», реальний опис у Book Details і першу
  cross-book навігацію (author shelf). Нічого не редизайнить.

**CM-агностичність (ключове рішення):** канонічний matching **уже реалізований**
(`packages/scrapers/src/canonical/match-canonical.ts`, працює online під час скрейпу через
`packages/api/src/pipeline/persist-listing.ts`). W9a **не** вводить нової CM-залежності, бо enrichment
зберігається на рівні **`provider_listings`**, а display-обкладинка/опис обираються **детерміновано**.
Деталі — §«CM Clarification» нижче.

## CM Clarification (нормативний блок)

Цей блок нормативний і має пріоритет над будь-яким іншим формулюванням у документі.

- **CM уже реалізований.** `match-canonical.ts` виконує ISBN-exact → нормалізований ключ → fuzzy
  (title 0.85 / author 0.80 / final 0.85) → conflict-розрізнення; працює online при скрейпі. Файл
  `docs/prd/canonical-matching.md` — чернетка-спека, а **не** статус коду.
- **W9a не потребує змін CM.** Жодних правок алгоритму, порогів, edge-cases чи злиття як вимоги W9a.
- **Enrichment зберігається на `provider_listings`**, **не** на `canonical_books`. Кожен листинг несе
  власну обкладинку та опис зі своєї product-сторінки.
- **Display cover/description обираються детерміновано** (правило — §9): пріоритет провайдера → перший
  непорожній серед кандидатів → placeholder fallback. Book Details бере з **усіх** листингів (включно з
  out-of-stock), Search — лише з in-stock.
- **W9a є CM-агностичним.** Обкладинки успадковують наявне групування CM, не залежать від злиття і не
  потребують його доопрацювання. Єдиний залишковий ефект — косметичний (§15, ризик щодо видимості наявних
  помилок злиття), не блокер.

## 2. Goals

- **Побачити.** Реальні обкладинки в Search Results, Book Details та «Бажанках» — список результатів стає
  полицею, яку *проглядають* очима, а не таблицею тексту.
- **Оцінити.** Реальний опис у Book Details — сторінка книги стає місцем, де *вирішують*, а не лише бачать
  ціну.
- **Перейти.** Author shelf на Book Details — перша cross-book навігація в продукті (книга → інші книги
  того ж автора).
- **Лишитись CM-агностичним.** Випустити W9a без жодних змін у канонічному matching (§CM Clarification).
- **Мінімум бекенду понад enrichment-скрейп.** Жодних нових ендпоінтів; author shelf працює на наявному
  `GET /api/search`. Нова робота на бекенді — лише two-field product-page enrichment + 2 nullable-колонки +
  наповнення вже наявних (нині `null`) полів DTO.
- **Зберегти заморожені патерни.** Search Results v1.0, Book Details v1.1, «Бажанки» v1.0 не редизайняться.

## 4. User Value

Дискавері — це не окрема сторінка, а **здатність побачити, оцінити й перейти**. W9a уперше робить продукт
*проглядальним і навігованим*:

- **Search Results → полиця, яку проглядають.** Користувач сканує обкладинки очима, а не вчитується в
  рядки тексту; швидше впізнає потрібне видання.
- **Book Details → місце рішення.** Обкладинка-hero + опис дають контекст, щоб вирішити «це та книга?»,
  перш ніж дивитись на ціни та офери.
- **Author shelf → перший крос-book перехід.** Від книги — до інших книг того ж автора; мінімальний, але
  справжній discovery-граф (книга→книга), на якому W9b/W9c добудовують повноцінні напрямки.
- **«Бажанки» → впізнаваність.** Реальні мініатюри роблять список збережених книг візуально скануваним.

## 5. Scope

W9a містить **рівно п'ять** елементів — нічого більше:

1. **Scraper enrichment.** Збагачувати листинги провайдерів **Yakaboo, Vivat, BookYe** (усіх, що мають
   реалізований скрапер) **лише** двома полями: `coverUrl` (з каталог-картки, де доступна) та `description`
   (з product-сторінки). (§8) BookClub — поза scope, доки не існує його скрапера.
2. **Persistence.** Зберігати enrichment **на `provider_listings`** (не на `canonical_books`). (§9)
3. **API enrichment.** Наповнити поля, що зараз `null`/відсутні: `SearchItemDto.coverUrl`,
   `BookDetailsDto.coverUrl`, `BookDetailsDto.description`, `WishlistItemDto.coverUrl`. (§10)
4. **Discovery UI wiring.** Реальні обкладинки в Search-картках, Book Details (hero), рядках «Бажанок»;
   реальний опис у Book Details. Наявні плейсхолдери лишаються fallback. (§11)
5. **Author shelf.** Best-effort полиця «Інші книги автора» на Book Details через наявний `GET /api/search`;
   graceful degradation. (§12)

- Опис, метадані, бейджі «Новинка»/сорт за популярністю **не** додаються (поза W9a).

### 6.2 Book Details v1.1 (Variant C)

- Ліва discovery-панель: **реальна обкладинка-hero** (220×320), коли доступна; інакше наявний градієнтний

## 7. Data Requirements

Два нові поля, обидва опційні / nullable. Джерела **різні**, тож і вартість різна (див. §8.1):

| Поле | Тип | Джерело | Нотатка |
|------|-----|---------|---------|
| `coverUrl` | `string \| null` | **каталог-картка** (URL зображення обкладинки, де доступна) | дешево — поле вже в HTML картки; стратегія джерела — §8.3 (B1); відсутність → placeholder |
| `description` | `string \| null` | **product-сторінка** (текст опису) | дорого — потребує per-book fetch; **санітизується до plain text** і **обрізається до безпечної межі** (~2–4k символів); HTML **не** рендериться |

- **Санітизація опису (нормативно).** Опис, витягнутий з product-сторінки, очищується до **plain text**:
  жодного HTML у вивід (захист від XSS — security rule «user input валідується на межі системи»). Markup
  product-сторінки нормалізується/видаляється при скрейпі або на межі API; у UI рендериться лише текст.
  Текст **обрізається до безпечної межі** (~2–4k символів) при скрейпі, щоб уникнути роздутих рядків.
- Гроші лишаються в копійках (`Money.amount`), UI форматує ÷100 → `N ₴` — W9a цього не змінює.
- W9a **не** додає жодних інших полів даних (видавець/мова/формат/рік/серія/популярність — поза scope).

## 8. Scraper Requirements

### 8.1 Enrichment — два різні шляхи (covers vs description)

Збагачення розщеплюється на **дешевий** і **дорогий** шлях; обидва — для **Yakaboo, Vivat, BookYe**:

- **Обкладинка (дешево).** `coverUrl` витягується з **уже наявного HTML каталог-картки** (`<img>` на
  картці) — **per-book product-page fetch не потрібен**. ~Нуль додаткових запитів.
- **Опис (дорого).** `description` лежить **лише на product-сторінці**, тож потребує **per-book
  product-page fetch pass** (N книг = N додаткових запитів).
- **F1 (covers) може випускатись незалежно від F2 (descriptions)** — і раніше, бо не залежить від дорогого
  product-page fetch.
- Розширити тип `RawProviderListing` (`packages/shared/src/types/provider.ts`) **двома опційними
  nullable-полями** (`coverUrl?`, `description?`). Існуючий контракт лишається сумісним.

### 8.2 Стійкість (errors collected, not thrown)

- Відсутність обкладинки/опису на сторінці **не ламає** скрейп листингу: поле лишається `null`, помилка
  збирається, а не кидається («errors collected, not thrown»).
- Зовнішні HTTP-запити — **з явним `timeout`** і **явним throttle (`delayMs`)**, **без прихованих
  retry-loop** (security rule). Per-book product-page запити (опис) мають враховувати rate-limits
  провайдерів: послідовний enrichment-pass після каталогу з агресивнішою затримкою; зупинятись на
  429/503. Відсутність обкладинки/опису **ніколи не валить** скрейп листингу.

### 8.3 Cover source strategy (рішення B1)

- **MVP — hotlink на зображення обкладинки з product-сторінки + placeholder fallback** (план R5).
- Зберігання / проксі-кеш обкладинок **відкладено** на пізніший крок (вартість + питання прав — §15).
- Стратегія лишається відкритим ризиком (§15, B1), але hotlink-with-fallback розблоковує MVP без
  інфраструктури зберігання.

### 8.4 Поза scope скрейпера

- Жодних інших полів (видавець / мова / формат / рік / серія) — це W9b.

## 9. Database Changes

- Додати **дві nullable-колонки** на таблицю **`provider_listings`**: `cover_url`, `description`.
- Колонки **на `provider_listings`, а НЕ на `canonical_books`** — це утримує W9a CM-агностичним
  (§CM Clarification). Кожен листинг несе власну обкладинку/опис із власної product-сторінки.
- Потрібна одна міграція (додавання nullable-колонок; беквард-сумісно).
- **Display-selection rule (детерміновано).** Коли під канонічною книгою кілька листингів, display-значення
  обкладинки/опису обирається детерміновано:
  1. кандидати сортуються за **пріоритетом провайдера** (фіксований порядок: `yakaboo` → `vivat` →
     `book-ye`), далі за ціною ↑;
  2. display = **перший кандидат із непорожнім полем**;
  3. якщо жоден листинг не має поля — `null` → UI показує placeholder/hint.
- **Набір кандидатів залежить від екрана:**
  - **Book Details** обирає з **усіх** листингів книги (включно з out-of-stock) — сторінка показується
    навіть коли всі офери `OUT_OF_STOCK`, тож обкладинка/опис не повинні зникати.
  - **Search** обирає лише серед **in-stock** листингів (книги без in-stock-оферів у видачі не з'являються).
- Правило має бути чистим і детермінованим (без random, без залежності від часу) — testing rule.

## 10. API Changes

Наповнити поля, що **сьогодні `null` або відсутні**; **без нових ендпоінтів**.

| DTO | Поле | Стан сьогодні | Після W9a |
|-----|------|---------------|-----------|
| `SearchItemDto` (`packages/api/src/search/dto.ts`) | `coverUrl` | **відсутнє** | додати `coverUrl: string \| null` |
| `BookDetailsDto` (`packages/api/src/books/dto.ts`) | `coverUrl` | hard-coded `null` | наповнити з display-selection |
| `BookDetailsDto` | `description` | hard-coded `null` | наповнити (sanitized plain text) |
| `WishlistItemDto.book` (`packages/api/src/wishlist/dto.ts`) | `coverUrl` | hard-coded `null` | наповнити з display-selection |

- Display-обкладинка/опис обираються **детерміновано** у відповідних мапперах
  (`packages/api/src/search/mapper.ts`, `packages/api/src/books/mapper.ts`, wishlist-маппер) за правилом §9.
- **Author shelf не потребує нового API** — використовує наявний `GET /api/search` за рядком автора.
- W9a **не** додає: пошук за ISBN, `sort`-параметр, suggest-ендпоінт, author/series-ендпоінти,
  popularity-поля (усе — W9b/W9c).

## 11. Frontend Changes

- Жодних змін розкладки, типографіки, бейджів, пагінації, сортування.

## 12. Author Shelf Requirements

- **Що це.** Полиця «Інші книги автора» на Book Details — перша cross-book навігація (книга → інші книги
  того ж автора).
- **Як працює.** Через **наявний** `GET /api/search` за рядком автора поточної книги (той самий підхід, що
  й Author Jump у W7a). **Near-zero backend**, без нових ендпоінтів.
- **Roster rule (нормативно).** Полиця будується з результатів `GET /api/search?q=<author>`, відфільтрованих
  до **точного збігу автора** — `author` поточної книги дорівнює `author` кандидата **без урахування
  регістру та з trim** (НЕ substring і не шумний збіг — це прецедент W7a Author Jump). Далі:
  1. виключити поточну книгу за `id`;
  2. виключити книги, уже показані Series shelf (`seriesIds`), якщо такі є;
  3. показувати полицю **лише коли після фільтрації лишилось N ≥ 2** книг; інакше — **повністю прихована**
     (без fallback-UI, без маскота).
- **Розкладка (frozen-дизайн).** Десктоп — 4-up грід BookCard (обкладинки 104×150); планшет — 3-up грід;
  мобільний — горизонтальний скрол-рейл. Деталі рендеру — у замороженому пакеті (нижче).
- **Заморожений дизайн-пакет (джерело рендеру).**
  `packages/web/design-import/discovery-w9/packages/web/design-import/discovery-w9a-author-shelf/` —
  **Author Shelf v1.0, frozen 2026-06-20**. **BookCard переюзається, не форкається.** Search Results,
  Book Details hero та «Бажанки» лишаються **недоторканими**. **Жодних додаткових discovery-поверхонь** W9a
  не потребує.
- **Лише author shelf.** Series shelf, «книга N із M», series jump — **W9b** (немає моделі серій).

## 13. Graceful Degradation Rules

- **Обкладинка відсутня** → наявний градієнтний placeholder; розкладка не змінюється.
- **Опис відсутній** → наявний partial-data hint («Опис ще не додано…»); сторінка не блокується.
- **Метадані відсутні** → «Уточнюємо…» (наявний стан; структуровані метадані — W9b).
- **Author shelf < 2 книг після точного-збігу фільтра** → полиця прихована повністю, без fallback-UI.
- **Partial-data ніколи не блокує сторінку** — Book Details лишається функціональним з ціною/оферами.
- **Сорт-чипи «Найпопулярніші»/«Новинки»** лишаються поза scope (немає сигналу) — у дусі W7a недоступний
  стан ховається/вимикається, а не імітується; повноцінно — W9c.

## 14. Acceptance Criteria

Конкретні й тестовані.

**Covers — Search**
- Search-результати показують **реальну обкладинку**, коли вона доступна.
- Search-результати показують **placeholder**, коли обкладинка відсутня.
- Розкладка картки та грід Search Results v1.0 — **без змін**.

**Covers — Book Details**
- Book Details показує **реальну обкладинку-hero**, коли доступна; інакше placeholder «Обкладинка».
- Layout Book Details v1.1 (Variant C) — **без змін**.

**Description — Book Details**
- Book Details показує **реальний опис**, коли доступний.
- За відсутності опису показується **partial-data стан** (hint), а не порожнеча; сторінка не блокується.
- Опис рендериться як **санітизований plain text** (жодного HTML / XSS-вектора у виводі).

**Covers — Wishlist**
- «Бажанки» показують **реальні обкладинки**, коли доступні; інакше placeholder.
- Layout «Бажанки» v1.0 — **без змін**.

**Author shelf**
- Author shelf рендериться **лише** коли після фільтра точного збігу автора (case-insensitive + trim) і
  виключення поточної книги / `seriesIds` лишилось **≥ 2** книг.
- Author shelf **повністю прихований** (без fallback-UI), коли лишилось < 2 книг.
- Збіг автора — **точний** (НЕ substring): кандидати з частковим/шумним збігом не потрапляють у полицю.
- Поточна книга не присутня у власній author shelf.

**Scraper / дані**
- Відсутність обкладинки чи опису на product-сторінці **не ламає** скрейп листингу (errors collected,
  not thrown); поле лишається `null`.
- Зовнішні запити мають явний timeout; немає прихованих retry-loop.

**CM / архітектура**
- **Жодних змін CM** не потрібно для W9a.
- Enrichment зберігається на `provider_listings`, не на `canonical_books`.
- Display-обкладинка/опис обираються **детерміновано** (однаковий вхід → однаковий вибір).

**Тести (project rule)**
- Нова бізнес-логіка (display-selection, санітизація, скрейп-парсинг) покрита тестами; coverage нових
  модулів ≥ 80%; тести детерміновані.

## 14a. Implementation Sequence (cheap-wins-first)

Порядок реалізації — спершу дешеві видимі перемоги (обкладинки в межах frozen-патернів), далі дорожчі шари:

1. **F1 — Covers enrichment.** Скрапер: `<img>` із каталог-картки (Yakaboo, Vivat, BookYe); 2 nullable-
   колонки на `provider_listings`; детермінований display-selection; наповнення API DTO.
2. **F4 — Search Result covers.** `coverUrl` у `SearchItemDto` + маппер; `ResultsGrid` пробрасує `cover`.
3. **F5 — Wishlist covers.** Реальна мініатюра у `WishlistRow` / мобільній картці замість placeholder.
4. **F2 — Description enrichment.** Per-book product-page fetch pass + санітизація до plain text + обрізання;
   Book Details уже рендерить опис із hint-fallback.
5. **F3 — Author shelf.** Порт замороженого пакета (§12) у `packages/web`; roster через `GET /api/search`
   з фільтром точного збігу автора; інтеграція останньою контентною секцією Book Details.

F1/F4/F5 не залежать від дорогого product-page fetch і можуть випускатись першими; F2/F3 — далі.

## 16. Future Development (поза W9a)

W9a — **фундамент**, на якому добудовуються повноцінні discovery-напрямки:

- **Серії** (модель, series shelf, «книга N із M», series jump) → W9b.
- **Author-сутність / канонікалізація** → надійні добірки та сторінки авторів (заміна best-effort author
  shelf) → W9b.
- **Структуровані метадані** (видавець / мова / формат / рік) → повний `<dl>`-грід Book Details → W9b.
- **Рекомендації** (series/author-related, «з цим читають») → W9b; semantic/ML → W9c.
- **Популярність** (сигнали, сорт «Найпопулярніші», коректний бейдж «Новинка») → W9c.
- **Discovery-лендинги / browse / стрічка** → W9b/W9c.

## 17. W9b Preview — Identity & Relationships (потребує бекенду)

- **Структуровані метадані** (видавець / мова / формат / рік) — scraper + колонки + API; приземлити
  `<dl>`-грід Book Details **повним разом із серіями**, а не напівпорожнім.
- **Канонічний matching уже реалізований**; W9b може **підвищувати точність** (edge-cases, ISBN-conflict,
  series-volume) за потреби — це **не** передумова W9a.
- **Author-сутність / нормалізація** → надійні сторінки-добірки автора (не substring).
- **Модель серій** (`seriesId`, порядок, тип) → series shelf, контекст «книга N із M», series jump.
- **Related-полиці** на основі серій + авторів — заміна best-effort author shelf із W9a.
