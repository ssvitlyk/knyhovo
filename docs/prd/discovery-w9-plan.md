# План: W9 Discovery (implementation-independent)

> Статус: **План — не PRD, не затверджено.** Код не пишеться до затвердження PRD (правило проєкту).
> Це аналітичний документ перед написанням PRD: розбір UX-специфікації, перевірка даних проти поточної
> архітектури, декомпозиція W9a/W9b/W9c, допущення, блокери, нестиковки, рекомендовані скорочення scope.
> Технологічний стек: **Next.js (App Router)** + **Fastify** + **Prisma/PostgreSQL**. Джерело UX —
> `packages/web/design-import/discovery-w9/` (frozen-патерни Search Results v1.0, Book Details v1.1,
> «Бажанки» v1.0, Price History v1.2.1).

## 1. Резюме UX-специфікації

W9 «Discovery» охоплює **три заморожені екрани** + інтегрований шар Price History. Самі екрани вже
реалізовані у фронтенді (`/search`, `/books/[id]`, `/wishlist`); W9 — це **discovery-шар даних і функцій
поверх них**, а не редизайн.

**1.1 Search Results v1.0 (frozen 2026-06-10).** Грід `BookCard` (2 кол. десктоп / 1 кол. мобільний).
Стани: idle · typing · loading · results · empty. Кожна картка: обкладинка · назва · автор · ціна
(serif accent) · стара ціна (muted strike) · книгарня (завжди muted) · один бейдж (Найкраща ціна → -N% →
Новинка). Сортування (чипи): найдешевші · **найпопулярніші** · **новинки**. Пагінація, мускот у empty.

**1.2 Book Details v1.1 (frozen, Variant C).** Дві панелі: ліворуч discovery (обкладинка 300×440 · назва ·
автор · опис · метадані) · праворуч decision (найкраща ціна · усі офери · вішлист). Метадані: видавець,
ISBN, мова, формат, **серія** («книга 1 із 8»), рік. Секції: Price History (W5, інтегровано) + **полиці
related**: «series shelf» першою, «author shelf» другою (4 кол. десктоп / горизонтальний скрол мобільний).
Стани: filled · loading · partial-data («Уточнюємо…») · unavailable (мускот + «Повідомити про наявність»).

**1.3 «Бажанки» (Wishlist) v1.0 (final freeze 2026-06-13).** Десктоп = Variant D, мобільний = Variant C
акордеон. Дві секції: «Книги зі знижками» (авто-промоут) → «Інші бажанки». Кожен рядок: обкладинка · назва ·
автор · **verdict-бейдж** (now/wait/high/data/out) + один honest reason-рядок · ціна · дельта · CTA
`[Деталі книги] [До книгарні]`. Лічильник **«Заощаджено з Knyhovo: N ₴»** (lifetime), **тижневий лист** від
Книговика, **collection-тег**, target price, tracking/alert toggles. Принцип: «Knyhovo шукає ціни. Книговик
радить.» Мускот лише в empty + first-book.

**1.4 Discovery-сигнали, що їх передбачає дизайн (наскрізно):** обкладинки (named «primary discovery
signal»), автори (як перехід і добірка), серії/колекції (полиці + контекст), популярність (сорт), related-
рекомендації (полиці series/author).

## 2. MVP scope — W9a: фундамент Discovery (covers + опис + author shelf)

> **Рішення (зафіксовано):** **Опція 4 — covers + опис + author shelf** (обґрунтування — §2.2). Структуровані
> метадані (видавець/мова/формат/рік) **відкладено до W9b**. W9a — це **фундамент дискавері**, а не набір
> окремих discovery-напрямків (чому саме «Discovery» — §2.3).

W9a поділено на **два чіткі шари**: спершу дані (Foundation), потім спосіб, у який ці дані стають
дискавері (UX Layer). Це той самий принцип, що й W7a: інтелект-шар поверх заморожених патернів, без
редизайну.

### Шар 1 — Discovery Data Foundation (бекенд: дані, яких раніше не було)

Збагачує модель книги двома полями, що їх система фізично не мала — обкладинка та опис. Це **перцептивний
субстрат** дискавері: без зображення список книг не можна *проглядати*, лише *читати*.

1. **Scraper enrichment** — витягувати з **product-сторінок** провайдерів (Yakaboo, BookClub) лише
   `coverUrl` та `description`. Розширити `RawProviderListing` (`packages/shared/src/types/provider.ts`)
   двома опційними nullable-полями; відсутність не ламає скрейп («errors collected, not thrown»).
2. **Schema** — додати 2 enrichment-колонки на **`provider_listings`** (CM-агностично — див. §2.2/R8/B2):
   `cover_url`, `description`. Міграція + правило вибору display-значення при кількох листингах (пріоритет
   провайдера / перший непорожній).
3. **API** — наповнити поля, що **зараз hard-coded null**: `BookDetailsDto.coverUrl` + `description`;
   `SearchItemDto.coverUrl`; `WishlistItemDto.coverUrl`. Display-обкладинка обирається детерміновано.

### Шар 2 — Discovery UX Layer (фронтенд: дані стають дискавері)

Перетворює нові дані на дискавері-поведінку в **уже наявних** екранах — нічого не редизайнить.

4. **Visual browse** — реальні обкладинки у `BookCard` (Search), hero Book Details, мініатюри «Бажанок»;
   рендер опису в Book Details (зараз плейсхолдер); partial-data «Уточнюємо…» / placeholder для відсутніх.
   Так список результатів стає **полицею, яку проглядають**, а не таблицею тексту.
5. **Author shelf (перша cross-book навігація, near-zero backend)** — полиця «Інші книги автора» на Book
   Details через **наявний** `GET /api/search` за рядком автора (як Author Jump у W7a). Це **єдиний
   discovery-перехід книга→книга** у W9a. Graceful degradation: ховається, якщо результатів мало/шумно.

**Явно НЕ входить у W9a** (це окремі discovery-*напрямки/destinations*, а не фундамент — усі винесені у
W9b/W9c):
- **Жанри / категорії** (немає сутності — B5-споріднене)
- **Тренди / «що зараз популярно»** (немає сигналу популярності — B4) → W9c
- **Популярні книги** (сорт за популярністю) → W9c
- **Популярні автори / лендинги авторів** (немає author-сутності — B6) → W9b
- **Рекомендації** («схожі», «з цим читають», semantic/ML) → W9b (series/author) / W9c (ML)
- **Окремі discovery-сторінки** (genre-/series-/author-landing, discovery-стрічка) → W9b/W9c
- Series shelf + «книга N із M», структуровані метадані (видавець/мова/формат/рік), verdict-движок
  «Бажанок», тижневий лист, лічильник заощаджень.

## 2.1. Чи потрібен CM для W9a? — НІ

**Канонічний matching уже реалізований** (`match-canonical.ts`, працює online при скрейпі). W9a **не**
вводить нової CM-залежності за умови:
- enrichment-поля зберігаються на рівні **`provider_listings`** (кожен листинг несе власну обкладинку з
  своєї product-сторінки), а не на `canonical_books`;
- API обирає display-обкладинку/опис **детерміновано** (пріоритет провайдера або найдешевший in-stock) з
  **placeholder fallback**.

Тоді обкладинки успадковують наявне групування CM, не залежать від злиття і не потребують його доопрацювання.
Єдиний залишковий ефект — косметичний: обкладинки роблять **наявні** помилки злиття видимішими (невідповідна
обкладинка помітніша за хибно згруповану ціну). Мітигація — детерміноване джерело + placeholder. **Висновок:
W9a можна випустити без жодних змін у CM.**

## 2.2. Найменший W9a, що дає видиму цінність — матриця опцій

Спільний для **всіх** опцій фіксований кошт: product-page scraping (B3) + рішення про джерело обкладинок
(B1). Він домінує — тому «covers only» платить повний кошт інфраструктури за одне поле (погана
кост-ефективність). Після того, як скрейпер уже ходить на product-сторінку, додати ще пару полів — дешево.

| Опція | User value | Складність | Scraper | Schema | API | Ризик |
|-------|-----------|-----------|---------|--------|-----|-------|
| **1 · Covers only** | Високий per-unit (візуально перетворює списки), але Book Details лишається голим; **немає discovery-дії** | Середня (фікс. кошт B1+B3 за 1 поле) | product-page fetch + 1 поле (cover URL) | +1 кол. | coverUrl у 3 DTO + display-rule | B1 домінує; інакше низький; погана кост-ефективність |
| **2 · Covers + опис** | Високий: ціль переходу стає інформативною | Середня | +опис (HTML→чистий текст) | +2 кол. | +description у BookDetails | +санітизація (XSS — лише plain text), варіативність розмітки |
| **3 · Covers + повні метадані** | Інкрементальний: добиває `<dl>` (видавець/мова/формат/рік), але **без серії** грід усе одно частковий | Середньо-висока | +4–5 полів, більше селекторів | +5–6 кол. | +поля у BookDetails | більше крихкості скрейпера, кожне поле окремо малоцінне |
| **4 · Covers + опис + author shelf** | **Високий + єдина реальна discovery-дія** (перехід до інших книг автора), near-zero backend | Середня | як опція 2 | як опція 2 (+2 кол.) | як опція 2 + author shelf через наявний `/api/search` | author shelf шумний (B6) → graceful hide; додатковий, низькоризиковий |
| **5 · Повний W9a** | Найвищий, повна історія | Висока | усі поля, макс. крихкість | ~6 кол. | усі DTO | найбільший кумулятивний QA + крихкість |

**Рекомендація — Опція 4 (Covers + опис + author shelf).** Це найменший scope, що **реально** заслуговує
назви «Discovery»: обкладинки = візуальна основа; опис = варта переходу ціль; author shelf = єдина
здійсненна *discovery-навігація* (книга → інші книги), майже без бекенду й з graceful degradation.
Структуровані метадані (видавець/мова/формат/рік) **відкласти й об'єднати з серіями у W9b**, щоб `<dl>`-грід
приземлився повним, а не напівпорожнім. **Опція 1 не рекомендована** попри «найменшість»: платить повний
фіксований кошт скрейпінгу за одне поле і не додає discovery-дії. Опції 3/5 — надлишкові для MVP.

## 2.3. Чому W9a — це «Discovery», хоча ще не вводить окремих discovery-напрямків

W9a **не** додає жодного окремого discovery-destination — ні жанрів, ні трендів, ні популярних книг/авторів,
ні рекомендацій, ні окремих лендингів. І все ж це саме **Discovery**, бо дискавері — це не окрема сторінка,
а **здатність побачити, оцінити й перейти**. W9a закладає всі три передумови, без яких будь-який майбутній
discovery-напрямок не має на чому стояти:

- **Побачити (covers).** Обкладинка — перцептивний субстрат проглядання. Поки книги — рядки тексту, грід
  результатів *читають*, а не *проглядають*. Обкладинки перетворюють Search/«Бажанки» на полицю, яку
  скануєш очима. Це фундамент будь-якої візуальної дискавері-стрічки чи лендингу — їх неможливо відрендерити
  без зображень. Дизайн недарма називає обкладинку «primary discovery signal».
- **Оцінити (опис).** Перехід до книги має сенс лише тоді, коли ціль інформативна. Опис робить Book Details
  місцем, де *вирішують*, а не лише *бачать ціну*. Без нього будь-яка рекомендація вела б у глухий кут.
- **Перейти (author shelf).** Перша **cross-book навігація** в продукті: від книги — до інших книг того
  ж автора. Це мінімальний, але справжній discovery-граф (книга→книга), а не просто прикрашений список. Усі
  складніші напрямки (серії, рекомендації, лендинги) — це **розширення цього самого графа**.

**Позиціонування одним реченням:** W9a — це **Discovery Foundation**, а не **Discovery Destinations**. Він
дає перцептивний шар (covers), варту переходу ціль (опис) і перший навігаційний примітив (author shelf), на
яких W9b (серії, автор-сутність, лендинги) і W9c (популярність, тренди, ML-рекомендації) добудовують
повноцінні discovery-напрямки. Назвати реліз «Discovery» чесно: він уперше робить продукт *проглядальним і
навігованим*, навіть якщо окремі discovery-сторінки приходять пізніше.

## 3. W9b та W9c

**W9b — Identity & Relationships (потребує бекенду):**
- **Структуровані метадані** (видавець / мова / формат / рік) — scraper + колонки + API; приземлити
  `<dl>`-грід Book Details **повним разом із серіями** (а не напівпорожнім, як було б у W9a).
- **Канонічний matching — вже реалізований**; W9b може **підвищувати точність** (edge-cases EC-6/EC-8,
  ISBN-conflict, series-volume) за потреби, але це **не** передумова W9a.
- **Author-сутність / нормалізація** → сторінки-добірки автора (надійні, не substring).
- **Модель серій** (`seriesId`, порядок, тип) → series shelf, контекст «книга N із M», series jump (W7b).
- **Related-полиці** на основі серій + авторів (заміна best-effort author shelf із W9a).

**W9c — Popularity & Recommendations (потребує інфраструктури сигналів / ML):**
- **Популярність** — збір сигналів (перегляди / частота пошуку / агрегати вішлиста) → сорт
  «Найпопулярніші», коректний бейдж «Новинка» (потребує `publicationYear`/дати релізу).
- **Semantic / ML рекомендації** — «що читати далі», схожість (перетин з `docs/prd/ai-discovery.md`,
  спорідненість з W7c).

## 4. Допущення

- A1. Три екрани вже реалізовані; W9 додає дані/функції, не редизайнить frozen-патерни.
- A2. Обкладинки/опис/серія/видавець фізично присутні на **product-сторінках** Yakaboo та BookClub
  (катáложні сторінки їх часто не містять) — підлягає підтвердженню на старті PRD.
- A3. Гроші скрізь — цілі **копійки** (`Money.amount`), UI форматує ÷100 → `N ₴` (узгоджено з W5).
- A4. «Бажанки»-verdict/тижневий лист/заощадження виводяться з price-history + alert-логіки; це окрема,
  пізніша робота, не блокує W9a-обкладинки.
- A5. Author shelf у W9a — best-effort через наявний search; за відсутності якісних збігів просто прихований.
- A6. W4 alerts API+UI існують; **движок тригерів/email — не реалізований** (потрібен для повного циклу
  «Бажанок», але не для W9a).

## 5. Блокери

- **B1 · Джерело й права на обкладинки.** Hotlink із Yakaboo/BookClub крихкий і може блокуватись; зберігання/
  проксі-кеш = вартість + потенційні питання авторського права. **Рішення про стратегію джерела — до PRD.**
- **B2 · Канонічний matching — НЕ блокер** (виправлено: CM **реалізований** і працює online під час
  скрейпу). W9a **не** вводить нової CM-залежності, **якщо** зберігати enrichment на рівні
  `provider_listings` (кожен листинг несе свою обкладинку), а API обирає display-обкладинку детерміновано
  (пріоритет провайдера / найдешевший in-stock) + placeholder fallback. Залишковий ризик суто косметичний:
  обкладинки роблять наявні помилки злиття CM **видимішими** (невідповідна обкладинка помітніша за хибно
  згруповану ціну), ніж сьогодні. Мітигація — детерміноване джерело + placeholder; **не** передумова W9a.
- **B3 · Product-page scraping.** Поточні скрейпери читають каталог; обкладинки/опис/серія — на сторінці
  книги → потрібен per-book запит: більше запитів, ліміти, таймаути (правило безпеки: явні timeout, без
  прихованих retry-loop).
- **B4 · Немає жодного сигналу популярності** ніде в системі → чипи «Найпопулярніші»/«Новинки» у frozen
  Search Results нефункціональні; бейдж «Новинка» не має джерела дати.
- **B5 · Серії не змодельовані** → series shelf, «книга 1 із 8», series jump заблоковані.
- **B6 · Автор не канонікалізований** (вільний рядок) → добірки автора ненадійні; author shelf через
  substring може давати шум.
- **B7 · Verdict / тижневий лист / заощадження «Бажанок»** потребують advisory-логіки + історії + движка
  тригерів (W4b email не готовий) — не існують сьогодні.

## 6. Перевірка залежностей даних проти поточної архітектури

| Дані | Стан сьогодні | Локація / нотатка |
|------|---------------|-------------------|
| Назва | ✅ Є | `canonical_books.title`, `SearchItemDto`, `BookDetailsDto` |
| Автор | ⚠️ Є як рядок | `canonical_books.author` (TEXT); немає author-сутності/ID/канонікалізації |
| Обкладинка | ❌ Немає | API `coverUrl: null` (hard-coded); скрейпер не витягує; немає колонки |
| Поточна ціна | ✅ Є | `provider_listings.price_amount` (копійки) |
| Стара ціна / знижка % | ⚠️ Похідне | окремо не зберігається; `change.percent` з price-history |
| Книгарня / наявність | ✅ Є | `provider`, `availability` (in/out/unknown) |
| ISBN | ✅ Є (nullable) | `canonical_books.isbn` (indexed, не unique); каталог часто без ISBN |
| Видавець / мова / рік | ❌ Немає | відсутні у схемі/API |
| Формат | ⚠️ Лише скрейпер | детектується, але не персиститься; ebook/audio відфільтровані |
| Серія / колекція | ❌ Немає | немає сутності; інколи в рядку title, не парситься |
| Опис | ❌ Немає | API `description: null` (hard-coded); скрейпер не витягує |
| Price history | ✅ Є | `GET /api/books/:id/price-history` (W5) повністю реалізований |
| Популярність | ❌ Немає | жодних перегляд/пошук/тренд-сигналів; search сортує лише за ціною |
| Рекомендації / related | ❌ Немає | немає зв'язків book↔book, «та сама серія/автор», алгоритму |

## 7. Чи реалізовні обкладинки / автори / колекції / популярність / рекомендації на наявних даних

- **Обкладинки — НІ на наявних даних.** Потрібен scraper enrichment + колонка + API + рендер + рішення про
  джерело (B1). Здійсненно як W9a (приймаємо backend-роботу).
- **Автори — ЧАСТКОВО.** Рядок автора існує; «Author Jump» (точний збіг) вже у W7a. Полиця «книги автора»
  через наявний search — best-effort у W9a. Надійні **добірки/сторінки автора — НІ** без канонікалізації
  (W9b).
- **Колекції / серії — НІ.** Немає сутності; не скрейпиться. Series shelf, «книга N із M», series jump →
  W9b (модель серій).
- **Популярність — НІ.** Жодного сигналу в системі. Сорт «Найпопулярніші»/«Новинки» нефункціональний → W9c
  (інфраструктура сигналів).
- **Рекомендації — НІ** (окрім best-effort author shelf). Немає зв'язків book↔book. Series/author-полиці →
  W9b; semantic/ML → W9c (перетин з `ai-discovery.md`).

## 8. Нестиковки в UX-специфікації

1. **Ціна:** моки показують цілі `₴`, реальний API — копійки; формат ÷100 не зафіксований у спеці.
2. **Обкладинки** скрізь плейсхолдери, але названі «primary discovery signal» — спека припускає дані, яких
   у бекенді немає.
3. **ISBN-пошук:** плейсхолдер SearchBar обіцяє пошук за ISBN, але `/api/search` шукає лише title/author і
   не повертає `isbn` (відомо з W7a).
4. **Мобільні секції «Бажанок»:** «акордеон, закритий за замовчуванням» vs «той самий двосекційний спліт, що
   й десктоп» — суперечність (чи секції згортаються на мобільному?).
5. **Verdict-тон:** exploration використовував `blue` для високої ціни; final freeze → `neutral` (muted).
6. **Мускот в unavailable:** рання версія — є; final freeze — **прибрано** (утилітарний стан).
7. **Перейменування секцій** (кілька ітерацій) — фінально лише «Книги зі знижками» / «Інші бажанки».
8. **«series» vs «collection»** вживаються взаємозамінно.
9. **`popularity`/`isNew`** присутні в моку BookCard, але без джерела в бекенді.
10. **Related-полиці** «серія перша, автор другий» — жодних із цих даних не існує.
11. **Лічильник заощаджень** — моки неоднозначні щодо періоду; freeze фіксує «за весь час».

## 9. Рекомендовані скорочення scope перед PRD

- **R1.** Прибрати/сховати чипи сорту «Найпопулярніші» та «Новинки» у W9a (немає сигналу) — graceful
  degradation у дусі W7a; повноцінно → W9c.
- **R2.** Series shelf + «книга N із M» + series jump → **W9b** (не W9a).
- **R3.** Recommendations / схожість → **W9c**.
- **R4.** Author shelf у W9a — лише best-effort через наявний search; прихований за шуму.
- **R5.** Обкладинки: **спершу зафіксувати стратегію джерела** (B1). Для MVP — рекомендую hotlink із
  fallback-плейсхолдером; зберігання/проксі винести у пізніший крок.
- **R6.** Метадані: скрейпити лише поля, надійно наявні на product-сторінці; відсутні → «Уточнюємо…»
  (partial-data стан спека вже підтримує).
- **R7.** «Бажанки» у W9a — лише підставити реальні обкладинки/метадані в наявні рядки; **verdict-движок,
  тижневий лист, лічильник заощаджень — пізніша хвиля** (потребують advisory-логіки + W4b email-движка).
- **R8.** Зберігати enrichment-поля на рівні **`provider_listings`** (не `canonical_books`); API обирає
  display-обкладинку/опис детерміновано (пріоритет провайдера або найдешевший in-stock) + placeholder
  fallback. Так W9a лишається **CM-агностичним** — CM уже реалізований і не є передумовою (див. B2).
